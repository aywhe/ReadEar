- 我在 Book 中添加的断点信息，dao 中的 insertPage 策略改为 Replace
当要从断点开始解析时，首先查询数据库中是否有断点信息，如果有，就从断点处开始，
这样不用每页都存储源页码了。
加入解析回调，每解析一个源文件的页码，就回调视情况保存断点信息

- 断点继续解析的功能，需要数据库存储最新的数据对应源文件的页码，然后一层层从TextManager->TextLoader->TextExtractor传参，
这样做以后，发现还需要 TextChunk 的 index 和 Page 中的 pageNumber 与源文件的页码对应
所以需要两个类都添加新的成员 originPosition, 同样数据库两个表也都要新增一列
考虑到，不用扫描版pdf的时候，解析速度还是很快的，断点继续解析的功能似乎没有用处
而在扫描版pdf中使用ocr识别文字，速度很慢，就很需要这个功能了，但是ocr的效果很不好，错字，漏字情况严重，似乎不需要继续开发
所以断点继续解析的功能在主分支不需要，ocr 分支后续看情况是否继续开发时再添加
还有 TextChunk 和 Page 的内容很多都是跨页（源文件的页码）的，断点继续很难对齐
解决了对齐问题的话，通常需要覆盖数据库中一些已经存在的 page，数据库存储又有问题：
如果 pageNumber 已经存在，在数据库中 insertPage 时会被忽略，因为策略是忽略，
如果 insertPage 策略改为 Replace 的话， 外部程序就要非常小心，不然会导致前部分大量的重复保存
导致外部程序总是要查询一下数据库再保存，增加了复杂度和时间开销，感觉不太划算
一边是大量的重复保存数据库，一边是重复的从文件解析文本

- MainActivity 中的 FileScreen 中实现文件列表拖动的功能时，使用了 org.burnoutcrew.composereorderable:reorderable:0.9.6
出现的问题是，拖动第0个文件时，列表会非常快的滚动，导致一下子跳过好多个文件，原因可能是对第0个文件的卡片坐标计算有问题
解决办法： 放弃了 org.burnoutcrew.composereorderable:reorderable:0.9.6， 改用 sh.calvin.reorderable:reorderable:3.0.0
虽然实现方法有些差别，但是后者就没有这个问题了

- ContentActivity 中 ContentScreen 调用 DraggablePlayButton， 回调函数 onShake 时 isSpeaking 一直都是 false
解决办法： DraggablePlayButton 中的包含 onDrag 的 pointerInput 的参数 Unit 改为 onShake, isSpeaking, onClick, onStop
不然 pointerInput 不会更新，导致 onShake 中的 isSpeaking 一直都是 false
